# CN-004: 集団抱持最小ルーブリック（ISS-34）

## 1. 概念の核心（一文定義）
個人の抱持を、対話・制度・外部化へスケールさせたときに観測すべき最小チェックリスト（[SCALE_JUMP]前提）を保持する概念ノート。

## 2. なぜこの概念が必要か
- 注: 作成元REPORTは現在 `codex/archive/deepdive/REPORT-collective-抱持-rubric.md` に配置。関連: `codex/archive/deepdive/REPORT-container-mapping.md`
- 個人の努力だけでは保持コストが回らない局面があるため、場（集団・制度）で保持を分散する設計が必要になる。
- ただしスケールジャンプは誤写のリスクが高いので、観測可能な最小ルーブリックに落とす必要がある。
- 本ノートは「組織的抱持（ISS-34）」の最小回答として位置づける。

> 個人7段階の写像ではなく、集団の観測可能性に合わせた簡易スペクトラム。

| 段階（集団） | 一言 | 主な兆候（3軸） | 典型リスク |
|---:|---|---|---|
| G1 | 即断・単純化 | 対話: 保留が出ない / 制度: 止められない / 外部化: 差分が残らない | 2-3（即応/早期収束）へ落ちやすい`[SCALE_JUMP]` |
| G2 | 擬似保留 | 対話: “検討してる感” / 制度: 手順はあるが更新が出ない / 外部化: 量だけ増える | 固着（反復の正当化）`[SCALE_JUMP]` |

| G4 | 生成的保留 | 3軸が連動し、保持コストが分散され、学習が螺旋する | 命令化/形骸化でG2へ反転`[SCALE_JUMP]` |

# §C: 個人7段階チェックリストとの対応表（最小）

> 目的: 個人指標を集団へ同一視しないまま、「どの個人兆候が集団で何として観測されるか」の翻訳表を置く。

| 個人（段階） | 集団での見え方（翻訳） | 観測の入口 |
|---:|---|---|
| 1（凍結） | 会議参加が消える/沈黙が増え、ズレ報告が消える | P-5, D-2 |
| 2（即応） | 反論・責任追及が増え、単純化圧が上がる | D-1, D-4 |
| 3（早期収束） | “結論が早い”が報酬化され、保留が出ない | P-3, D-1 |
| 4（擬似抱持） | 手順/資料は増えるが差分が出ず、同じ議論が戻る | D-3, E-2, P-6 |
| 5（関係依存） | 特定個人（上司/番頭/ファシリ）への依存が強まり、場がその人で上下する | D-6, P-1 |
| 6（good-enough） | 個人の保留運用が、対話・制度・外部化に“接続”し始める | D-1, P-4, E-1 |
| 7（生成的） | 個人ではなく場として保留が守られ、保持コストが分散される | P-1, D-1, E-4 |

# §D: 統合と限界

## D.1 本ルーブリックの適用範囲

- 適用対象: 会議、1on1、レビュー、ふりかえり等、「判断/更新/保留」が発生する場面。
- 適用しない: 個人の身体兆候の推定（集団では観測困難）。ここでは扱わない。

## D.2 `[UNCERTAIN]`残量レポート

- 外部化（共有物）が“更新の保証”になるわけではない（擬似更新の可能性）`[UNCERTAIN][SCALE_JUMP]`
- 「制度がある」だけで抱持が成立するわけではない（命令化/形骸化の反転）`[UNCERTAIN][SCALE_JUMP]`

## D.3 ISS-34の残課題（将来タスク候補）

1. 役割分布モデル: 集団は“段階の分布（個人差×役割）”を持つという前提の精密化（どの役割がどのContainerを担うか）。`[SCALE_JUMP]`
2. 失敗モード分類: G2（擬似保留）の型（制度硬直/資料増殖/責任回避など）を複数類型化する。
3. 観測指標の検証: 実際の会議ログ/運用ログで、チェック項目が現実に識別力を持つかを検証する（opsレベルの検証）。
4. 個人↔集団の接続条件: 個人の段階6が場によって支えられ内在化する、の“条件セット”を明文化する（EV-DP-002のスケール版）。
